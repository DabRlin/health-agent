## 二、待办事项

### P0 — 高优先级（直接影响答辩质量）

#### 1. Agent 工具链扩充

**现状**：7 个工具（get_health_metrics / get_health_trend / run_risk_assessment / get_user_profile / add_health_metric / get_health_knowledge / analyze_exam_report）

**可扩充方向**：

| 工具名 | 功能 | 价值 | 工作量 |
|--------|------|------|--------|
| `compare_metrics` | 横向对比多项指标（如血压+心率联合分析） | Agent 能做更深入的综合分析 | 中 |
| `get_health_summary` | 生成阶段性健康总结（周报/月报） | 丰富 Agent 输出，用户体验好 | 中 |
| `update_user_profile` | 通过对话更新健康档案字段 | 对话式档案维护，闭环体验 | 低 |
| `list_exam_reports` | 列出所有体检报告，支持按时间范围筛选 | 当前只能取最新一份，扩展查询能力 | 低 |

**建议**：优先做 `get_health_summary` 和 `update_user_profile`，前者丰富 Agent 分析深度，后者补齐对话式交互闭环。扩充到 9 个工具即可，不宜太多（LLM 工具选择准确率随工具数增加而下降）。

#### 2. 代码质量、错误提示、安全性整改

**已知问题清单**：

| 文件 | 问题 | 修复方案 |
|------|------|---------|
| `auth_service.py` | 部分 `traceback.print_exc()` 残留 | 替换为 `logging.exception()` |
| `user_service.py` | `str(e)` 直接暴露异常堆栈给前端 | 已部分修复，需全面排查 |
| `exam.py` 路由 | 上传失败时异常信息泄露 | 已部分修复，需确认 |
| 全局 | 无统一异常处理中间件 | 添加 Flask `errorhandler` 统一捕获 |
| `app.py` | CORS OPTIONS 预检处理 | 已添加，需验证 |
| `agent_tools.py` | `execute_tool` 中 `str(e)` 暴露内部错误 | 改为通用错误提示 |
| JWT | token 无刷新机制，过期后直接踢回登录 | 可加 refresh_token（视时间） |

**建议**：集中做一轮安全性扫描，耗时约 1-2 小时。优先处理异常信息泄露和统一错误处理。

#### 3. 论文初稿编写

**系统架构相关素材已齐备**：
- 技术栈：Vue 3 + Flask + SQLite + ReAct Agent + RAG + 4 个医学模型

- 系统特色：自建 Agent 而非调用第三方、RAG 知识库、流式交互

**建议**：可与开发并行推进，先搭论文框架（绪论、相关技术、系统设计、实现、测试），代码改进可作为论文"系统实现"章节的内容。

### P1 — 中优先级（提升系统完整度）

#### 4. 管理员界面

**需求**：
- 管理员鉴权（Account 表新增 `role` 字段，值为 `user` / `admin`）
- 管理员可查看/管理用户列表（禁用、查看档案）
- 管理员可管理 RAG 知识库（查看索引状态、触发重建、上传新文档）
- 管理员可查看系统统计（用户数、问诊次数、评估次数等）

**实施方案**：
1. 数据库：Account 表加 `role` 字段，默认 `user`，种子数据创建一个 admin 账户
2. 后端：新增 `admin` 路由蓝图 + `@admin_required` 装饰器
3. 前端：新增 `/admin` 路由和 AdminDashboard.vue 页面，侧边栏根据角色动态显示

**工作量**：约 1-2 天

#### 5. 趋势图表空缺日期 + 预测算法升级

**空缺日期问题**：
- 后端 `trend_service.py` 只返回有记录的日期，空缺天被直接连线
- `TrendAnalyzer.linear_regression` 用序列索引作 X 轴，不感知日期间距
- 当前种子数据连续 30 天无影响，真实使用时会有偏差

**预测算法升级**：
- 当前：移动平均 + 线性回归
- 升级为：Holt-Winters 指数平滑（无需新依赖，纯 Python 可实现）
- 论文引用价值高，改动范围限于 `trend_analysis.py`

**建议**：两者可合并处理。先改 `linear_regression` 接收实际天数差作为 X 轴（修复空缺问题），再将预测方法升级为 Holt-Winters。

#### 6. 健康数据页面支持编辑已有记录

**现状**：只有"添加"入口，无法修改已录入的数据

**方案**：
- 在趋势图表下方添加近期记录列表
- 每条记录支持点击编辑（复用添加弹窗，预填已有值）
- 后端已有每日唯一制逻辑，编辑本质上就是重新提交当天数据

**工作量**：约 2-3 小时

#### 7. 健康报告列表页

**现状**：README 路线图标记已完成，但路由中无 `/reports` 路由，侧边栏也无入口

**需确认**：是否已合并到首页或其他页面？如果没有，需要补建或从路线图中移除。

### P2 — 低优先级（锦上添花，视时间决定）

#### 8. UI 设计美化（黑白主色调 + Material Design 3）

**评估**：
- 当前为 Meta 风格设计系统，整体一致性较好
- 大规模改色调和设计语言涉及全部页面，工作量大（3-5 天）
- 对答辩加分有限

**建议**：如果时间充裕可做局部调整（如暗色模式切换），不建议全面重构。

#### 9. Dockerfile 容器化部署

**评估**：
- 前后端分离架构适合容器化，Dockerfile 编写本身不复杂
- 需要处理：SQLite 数据卷挂载、环境变量注入、前端构建、Nginx 代理
- 对论文"系统部署"章节有加分

**方案**（如果做）：
```
docker-compose.yml
├── backend/Dockerfile    (Python + Flask)
├── frontend/Dockerfile   (Node build + Nginx)
└── nginx.conf            (反向代理)
```

**建议**：收工前花半天写好即可，不影响主线开发。
