## 一、ReAct Agent 架构——自主工具调用与推理

### 核心亮点

本项目的智能问诊**没有采用简单的"问答对话"模式**，而是实现了完整的 **ReAct（Reasoning + Acting）循环**：LLM 自主决策是否调用工具、调用哪个工具、如何利用工具结果继续推理，最终给出有数据支撑的专业回答。

### 技术实现细节

**双模式流式策略（关键优化）**

这是本项目最核心的工程难点。OpenAI Function Calling 存在一个根本矛盾：`stream=True` 时才能实现逐字输出，但工具调用需要等完整的 JSON arguments 拼接完成后才能执行。

本项目的解决方案：

```
第一轮（stream=True + tools）：
  - 若 LLM 直接回答 → 文本 chunk 实时推送给前端（真正逐字流）
  - 若 LLM 要调用工具 → content 为空，tool_calls 分块收集并拼接

后续轮（stream=False）：
  - 判断是否继续调用工具
  - 最终回答时切回 stream=True 保证流式体验

最终兜底：超过 MAX_TOOL_ROUNDS(5) 后强制流式输出
```

这个策略在"第一次无工具调用"这个最常见场景下，实现了**零延迟的真正流式输出**；同时对"需要工具调用"的场景无缝切换。

**多工具并发支持**

流式收集 tool_calls 时按 `index` 字段分槽拼接，支持 LLM 在一次响应中并发调用多个工具（如同时查询健康指标和健康趋势）。

**科室级工具过滤**

不同科室暴露不同工具集（`TOOLS_BY_DEPARTMENT`）：皮肤科仅暴露知识库工具，避免模型调用无关工具；全科暴露全量 7 个工具。这既减少了无效工具调用，也提升了科室专业性。

## 二、多科室专科化设计

### 亮点

系统设计了 4 个专科门诊（全科/心血管科/内分泌科/皮肤科），每个科室有：
- **独立的 System Prompt**：包含科室擅长领域说明、可用工具说明、专科注意事项
- **独立的工具集**：皮肤科不提供体征查询工具，避免答非所问
- **独立的 RAG Collection**：检索时优先命中该科室专项文档片段

这种设计使得同一个 AI 模型在不同科室表现出完全不同的专业特性，是工程层面对"大模型专科化"的一个实践。

## 三、RAG 知识检索——向量召回 + 精排

### 亮点

知识库来源于权威医学文献《默克家庭诊疗手册》（3.5MB 中文版），采用两阶段检索：

```
向量召回（bge-m3 Embedding）→ Top-20 候选
         ↓
精排重排（bge-reranker-v2-m3）→ Top-3 最终
```

**双阶段的意义**：纯向量检索召回精度受限于 embedding 空间的局限性，bge-reranker 是专门为语义相似度打分训练的 cross-encoder 模型，精排后的结果在语义相关性上显著优于单纯向量检索。

**降级链路**：RAG 索引未构建 → 结构化 SQLite 知识库 → 确保系统在无向量索引时依然可用。

**科室路由**：Agent 调用知识库工具时，自动将 `department` 参数映射到对应 Collection，实现检索的科室专注性。

## 四、多模态 VL 模型集成

### 4.1 皮肤科图像分析

皮肤科支持用户上传皮肤图片，利用 **Qwen2.5-VL-72B** 进行图像理解，输出：
- 皮损外观描述（形态、颜色、分布）
- 2-3 种可能的皮肤状况参考
- 护理与就医建议

图像分析结果以"前置 assistant 消息"的形式注入对话上下文，使后续的文字对话可以自然引用图像内容，实现多模态信息融合。

### 4.2 药品说明书 OCR

用 **Qwen2.5-VL-72B** 对药品说明书图片进行结构化提取，直接输出 JSON 格式的结构化用药信息：用药时间、剂量、餐食关系、禁忌、不良反应等。相比传统 OCR + 后处理的方式，VL 模型能够理解版面结构，准确率更高。

### 4.3 体检报告 OCR

使用 **DeepSeek-OCR** 专用模型进行体检报告图片的文字提取，再交由 GLM-4.7 进行结构化解析，形成标准化的指标列表，供 Agent 直接调用分析。

## 五、四大医学评估模型的工程实现

本项目将四个国际权威医学风险评估工具以纯 Python 实现，无需外部医学计算库：

### Framingham Risk Score（心血管）

基于 Framingham Heart Study 的 Cox 比例风险模型，对男女分别建立评分表，综合年龄、胆固醇、血压、吸烟、糖尿病等因素计算 10 年心血管事件风险概率。

### FINDRISC（糖尿病）

Finnish Diabetes Risk Score，基于 8 个生活方式和体征因素的积分量表，0-26 分对应从极低到极高的 10 年糖尿病发生风险。

### 代谢综合征评估（IDF/NCEP 联合标准）

依据 IDF（国际糖尿病联盟）与 NCEP ATP III 标准，检测腰围、甘油三酯、HDL、血压、空腹血糖五项指标，满足标准数量决定是否诊断为代谢综合征及其严重程度。

### FRAX®（骨质疏松）

世界卫生组织骨折风险评估工具，无需骨密度（BMD）的简化版本，基于年龄、性别、BMI、骨折史、激素使用史、饮酒吸烟等评估骨质疏松骨折风险。

**缺失数据透明化**：四个模型均实现了缺失字段检测逻辑，当用户未填写某项必要数据时，系统用合理默认值计算并在结果中明确标注"部分数据使用默认值"，保证评估的可信度说明。

## 六、自动健康标签系统

系统在用户更新健康档案时，自动根据 25 条规则引擎评估并同步健康标签：

- 规则以纯 lambda 函数声明，维护简便
- 标签分为"系统标签"（自动计算）和"用户标签"（手动添加）
- 每次档案更新后全量重算，删除已不满足条件的旧标签，添加新触发的标签
- 标签类型分 `positive`（绿）和 `warning`（黄/橙），直观呈现健康状态

这套机制使用户的健康状况变化能实时反映在首页仪表盘，形成动态健康画像。

## 七、前端流式渲染与"思考气泡"交互

### SSE 流式接收

前端 `api/index.js` 的 `sendMessageStream` 方法使用 Fetch API + ReadableStream 实现 SSE 解析，支持：
- `chunk`：逐字拼接 AI 回复
- `thinking`：在消息列表中显示"思考中"气泡（附带工具调用描述文字）
- `done` / `error`：流结束处理

### Markdown 实时渲染

AI 回复内容通过 `marked.js`（支持 GFM + 换行）实时渲染为 HTML，支持表格、列表、粗体、代码块等格式，保证医学内容的可读性。

### 思考气泡设计

当 Agent 调用工具时，前端显示一个带有 CPU 旋转图标的"思考气泡"，并展示当前工具调用的语义说明（如"正在分析健康趋势数据..."），让用户感知 AI 的推理过程，提升交互体验。

## 八、安全与工程质量

### JWT 认证
- 使用 PyJWT 签发 Token，服务端无状态校验
- `@login_required` 装饰器统一鉴权，管理员接口额外检查 `role=admin`
- 401 时前端自动清除 token 并跳转，防止长时间 token 残留

### Agent 数据录入保护
- `add_health_metric` 工具的 System Prompt 明确要求"调用前需先向用户确认数值"
- 药品说明书提取结果先展示给用户确认再入库
- 防止 AI 自主写入错误数据

### 工具调用上限
- `MAX_TOOL_ROUNDS = 5` 防止 Agent 无限工具调用死循环
- 超过上限强制生成最终回复，保证请求必然结束

### CORS 配置
后端统一在 `after_request` 钩子中添加 CORS 头，前后端分离开发无跨域问题。
