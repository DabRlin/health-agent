## 一、多科室门诊体系

### 建议内容
将现有的单一智能问诊界面，改造为「先选科室 → 进入对应门诊」的模式。

### 现状
- 单一 `SYSTEM_PROMPT`，通用健康助手
- 单一 RAG 知识库（默克家庭诊疗手册，collection: `merck_manual`）
- 工具链全量注入（7 个工具全部可用）
- 问诊范围提示通过 prompt 中「边界意识」条款泛化约束

### 改造方案

#### 1. 科室划分（建议初期 4-5 个）
| 科室 | 说明 | 核心工具 | 知识库 |
|------|------|----------|--------|
| 内科 / 全科 | 通用症状、慢病管理 | 全量 | 默克手册（现有） |
| 心血管科 | 血压、心率、胸痛、心电图 | health_metrics / trend / risk(Framingham) | 心血管专项文献 |
| 内分泌科 | 血糖、糖尿病、甲状腺 | health_metrics / trend / risk(FINDRISC) | 糖尿病指南 |
| 皮肤科 | 皮疹、图像分析 | 视觉模型（新增）/ health_knowledge | 皮肤病图谱 |
| 药学咨询 | 说明书解读、用药提醒 | OCR + 用药提醒工具（新增） | 药品说明书库 |

#### 2. 技术影响评估
- **System Prompt**：每个科室独立一份，放在 `agent_service.py` 的字典中，按 `department` 参数选取，**工作量小**
- **工具链过滤**：`TOOLS_SCHEMA` 改为按科室过滤，`execute_tool` 加科室白名单校验，**工作量小**
- **RAG 多知识库**：`RAGService` 目前写死单个 collection，需改为按科室传入 `collection_name`，**工作量中**；知识库文本资料需另外收集
- **前端选科室页**：新增科室选择页/组件，进入问诊时携带 `department` 参数，**工作量中**
- **数据库**：`Consultation` 表需加 `department` 字段，**工作量小**

#### 3. 所需新知识库文本（待收集）
- 心血管：ACC/AHA 指南摘要、高血压防治指南
- 内分泌：中国糖尿病防治指南、甲状腺疾病规范
- 皮肤科：常见皮肤病图文资料（文字部分）
- 药学：常用药物说明书结构化文本

## 二、视觉模型接入（皮肤科图像分析）

### 建议内容
接入视觉模型，允许用户上传图片，以皮肤科为主要场景。

### 扩展场景（其他可能）
- **皮肤科**：皮疹、痣、伤口辨别 ✅ 最明确
- **骨科/康复**：X 光片辅助解读（需声明仅辅助参考）
- **眼科**：眼底图、结膜充血初筛
- **体检报告**：目前已有 OCR 解析体检报告，可以复用视觉能力增强

### 技术方案
SiliconFlow 支持视觉模型（如 `Qwen2.5-VL-7B-Instruct`），兼容 OpenAI 的 `vision` 格式，与现有 `llm_client.py` 同一 API base，接入成本低。

```
用户上传图片
  → 前端 base64 编码 或 后端存储后取 URL
  → 构造 vision message（content 为 text + image_url 数组）
  → 注入到问诊上下文，触发视觉模型分析
  → 结合 RAG 知识库给出参考意见
```

### 影响评估
- **后端**：`agent_service.py` 支持 vision message 格式，皮肤科 system prompt 加图像引导语，**工作量中**
- **前端**：Consultation.vue 增加图片上传入口（仅皮肤科科室显示），**工作量中**
- **存储**：图片临时 base64 或存 OSS/本地，需决策（临时 base64 最简单）
- **合规**：在 UI 上明确标注「AI 辅助参考，不构成诊断」

## 三、用药管理（OCR 说明书 + 通知提醒）

### 建议内容
用户拍摄/上传药品说明书 → OCR 提取关键信息 → 结构化存储 → 生成用药提醒通知。

### 复杂度拆解

#### 3.1 OCR 信息提取
| 提取字段 | 难度 | 说明 |
|----------|------|------|
| 药品名称 | 低 | 说明书标题 |
| 适应症 | 中 | 文本段落提取 |
| 用法用量 | 高 | 格式多样（成人/儿童/体重计算） |
| 用药频率 | 高 | 每日X次、每隔Xh、饭前/饭后 |
| 剂型 | 中 | 口服/外用涂敷/注射/滴眼 |
| 禁忌症 | 中 | 文本段落提取 |
| 有效期 | 低 | |

技术方案：视觉模型（同上）直接解析说明书图片为结构化 JSON，比传统 OCR + NLP 更准确，**可复用视觉模型能力**。

#### 3.2 数据模型
新增 `Medication` 表：
```
id, user_id, name, dosage, frequency, route(口服/外用/注射),
start_date, end_date, reminder_times(JSON), notes, created_at
```

#### 3.3 通知提醒
- 前端方案：浏览器 Notification API（仅前台）
- 后端方案：服务端定时任务（APScheduler）+ WebSocket/SSE 推送
- 推荐：前期用浏览器本地提醒（localStorage + setInterval），够用且工作量小

### 影响评估
- **工作量大**，涉及新数据模型、OCR 流程、提醒逻辑
- 建议分两期：
  - **一期**：OCR 提取 + 存储 + 展示（无提醒）
  - **二期**：用药提醒通知

## 四、综合优先级与建议开发顺序

| 优先级 | 功能模块 | 工作量 | 价值 | 建议 |
|--------|----------|--------|------|------|
| P0 | 多科室 System Prompt + 工具链过滤 | 小 | 高 | 立即做 |
| P0 | 数据库 + 前端科室选择入口 | 中 | 高 | 立即做 |
| P1 | RAG 多知识库支持 | 中 | 高 | 收集资料后做 |
| P1 | 皮肤科视觉模型接入 | 中 | 中 | RAG 之后 |
| P2 | 用药 OCR 提取 + 存储 | 大 | 中 | 一期只做提取 |
| P2 | 用药提醒通知 | 大 | 中 | 二期 |
| P3 | 其他科室知识库文本收集 | 取决于资料 | 高 | 持续进行 |
